# R2 Export Prep Checklist

## Цель R2 Export

Подготовить безопасный и предсказуемый контур для пользовательской выгрузки данных через команду `/export` в Telegram, без изменений Core и без запуска кодовой реализации на текущем шаге.

## Not touching Core

Этот документ фиксирует, что в рамках R2 Export Prep:

- не затрагиваются Tier-1 frozen файлы;
- не изменяются Core-компоненты;
- не изменяются FSM, schema, invariants;
- не добавляются миграции, тесты, воркеры, stub-команды;
- не выполняются изменения БД и runtime-логики.

## Scope и границы

В scope текущего шага входит только:

- формализация UX-контракта `/export`;
- формализация лимитов и audit-требований на концептуальном уровне;
- список данных и артефактов, необходимых для следующего шага реализации;
- проверочный checklist и acceptance criteria.

Вне scope:

- любая реализация (код, SQL, миграции, тесты, интеграции, деплой).

## Разрешено сейчас

- Документирование требований и ограничений.
- Согласование терминов и ожидаемого поведения с точки зрения пользователя.
- Подготовка безопасного плана выполнения для будущей реализации.

## Запрещено сейчас

- Кодовые изменения любого уровня.
- Любые изменения в Core / Tier-1 / FSM / schema / invariants.
- Добавление таблиц, логирования в БД, миграций.
- Изменение CI, Iron Fence, governance-пайплайна.
- Догадки о внутреннем устройстве Core и привязка к незафиксированным внутренним деталям.

## UX-контракт команды `/export`

### Формат команды

- Базовый вызов: `/export <category>`
- `<category>` — обязательный параметр категории выгрузки.

### Ожидаемое поведение (UX)

1. Пользователь отправляет `/export <category>`.
2. Система подтверждает принятие запроса понятным сообщением.
3. При успешной подготовке выгрузки пользователь получает результат в ожидаемом формате (CSV/Excel — определяется на этапе реализации).
4. При некорректном вводе система возвращает короткую подсказку по правильному формату команды.
5. При превышении лимита система сообщает, когда можно повторить запрос.

### UX-сообщения (концепт)

- Успех принятия: "Запрос на экспорт принят."
- Неверный формат: "Используйте: /export <category>"
- Лимит: "Лимит запросов исчерпан. Повторите позже."
- Ошибка выполнения: "Экспорт временно недоступен. Попробуйте позже."

## Rate Limit Policy (концепт)

- Лимит вводится на пользователя/чат для защиты от спама и перегрузки.
- Политика должна быть простой и предсказуемой.
- При срабатывании лимита пользователь получает явное уведомление с ожиданием повторной попытки.
- Финальные числовые пороги фиксируются отдельно перед реализацией (после подтверждения стабильности Iron Fence / CI).

## Audit Trail Requirements (концепт)

Для каждого запроса `/export` в будущем должны фиксироваться:

- кто запросил (идентификатор пользователя/чата);
- что запросил (категория, формат);
- когда запросил (timestamp);
- результат (accepted / limited / failed / delivered);
- correlation/trace идентификатор для трассируемости.

Принципы:

- минимально необходимые данные;
- без хранения лишнего чувствительного контента;
- запись должна быть пригодна для операционного разбора и ретроспективы;
- концептуальная точка хранения audit trail: `rev_export_logs`.

## Правило чтения данных для экспорта

- Чтение данных для экспорта выполняется только через **read-only views**.
- Прямые JOIN с таблицами Core запрещены.

## Данные, которые нужны потом для реализации

Перед стартом кодовой реализации должны быть согласованы:

- список допустимых категорий для `/export`;
- финальный формат(ы) выгрузки (CSV, Excel или оба);
- точные лимиты rate limit (порог и окно);
- правила авторизации (кто имеет право на экспорт);
- SLA/таймаут на подготовку выгрузки;
- правила именования файлов;
- политика хранения и доступности экспорт-файлов;
- перечень полей audit trail (включая привязку к `rev_export_logs`);
- ожидаемые тексты пользовательских сообщений (RU);
- перечень read-only views, являющихся единственным источником данных для выгрузки.

## Checklist реализации (без кода)

1. Подтвердить, что Iron Fence и CI стабильны и разрешают переход к кодовым шагам.
2. Утвердить финальный UX-контракт `/export`.
3. Утвердить список категорий и форматы выгрузки.
4. Утвердить rate limit policy (числа и окно).
5. Утвердить требования audit trail (поля и статус-модель, целевой журнал `rev_export_logs`).
6. Утвердить policy по доступу (auth) к команде `/export`.
7. Утвердить список read-only views для экспорта и зафиксировать запрет на прямые JOIN с Core.
8. Зафиксировать acceptance criteria в задаче реализации.
9. Только после этого запускать отдельную фазу реализации.

## Acceptance Criteria

Документ считается готовым, если:

- содержит только docs-only спецификацию;
- явно фиксирует "Not touching Core";
- не содержит кода, SQL, миграций, тестов, внутренних технических догадок;
- описывает UX-контракт `/export`, rate limit, audit trail и checklist;
- содержит правило: экспорт читает данные только из read-only views, без прямых JOIN с таблицами Core;
- даёт однозначную основу для безопасного старта реализации на следующем шаге.
