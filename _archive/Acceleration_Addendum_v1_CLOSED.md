## 2) Финальный состав ускоряющих механизмов (без overengineering)

Ниже — финальный набор механизмов (с учётом Opus v2 + критики Gemini + финальных микро‑закрытий “gap”).

### M1 — Tier‑1 Hash Lock (абсолютная защита Freeze)
**Цель:** гарантировать, что Tier‑1 файлы не менялись вообще (даже комментарии/whitespace).  
**Где:** CI (GitHub Actions).  
**Важно:** учесть **CRLF vs LF** (Windows локально ↔ Linux в CI). Хэш считается по **нормализованному содержимому** (удаляем `\r`).

**Покрывает:** дрейф Tier‑1, “случайный” AI‑edit, незаметные правки.

---

### M2 — Ruff в CI + pre‑commit (без красной стены)
**Цель:** ловить “глупые” ошибки до ревью и до pytest.  
**Правило внедрения:** стартуем в **мягком режиме**, чтобы не получить 1000 ошибок на старом коде:
- либо проверяем только изменённые файлы,
- либо ограничиваем набор правил и постепенно ужесточаем.

**Покрывает:** стиль/мёртвый код/ошибки импорта/часть багов, ускоряет цикл фиксов.

---

### M3 — Boundary Grep (ультралёгкий “Guardian” без AST)
Это **не** полноценный линтер инвариантов. Это 3–4 сверхконкретных проверки, которые дают 90% пользы при ~0% поддержки.

#### M3a: запрет DML по reconciliation_* из Tier‑3 Python
Ловим `INSERT/UPDATE/DELETE reconciliation_…` в `ru_worker/`, `side_effects/`, `webhook_service/`, `cli/`.

#### M3b: запрет raw‑DML по бизнес‑таблицам мимо Tier‑2 атомиков
Ловим прямые `UPDATE/INSERT` в ключевые бизнес‑таблицы внутри Tier‑3 (примерный минимум):
- `order_ledger`
- `shipments`
- `payment_transactions`
- `reservations`
- `stock_ledger_entries`
*(список уточняется по фактической архитектуре, но цель понятна: Tier‑3 не должен “хирургически” менять core‑данные)*

#### M3c: запрет импортов Tier‑1 frozen domain‑модулей из Tier‑3
Запрещаем `import domain.reconciliation_*`, `domain.structural_checks`, `domain.observability_service` из Tier‑3.

> Важно: отдельно запретить и `domain.reconciliation_alerts`, чтобы Tier‑3 не мог вызвать `emit_alert()` “в обход” sweeper’а.

#### M3d (необязательная, но полезная): trace_id в воркерах
Grep “по сигнатуре” хрупок (переносы строк). Вместо этого проверяем **факт извлечения trace_id из payload** в side_effect workers.

**Покрывает:** самый дорогой класс ошибок — нарушение границ Core и “инвариантов владения” без тяжёлой инфраструктуры.

---

### M4 — Migration DDL Guard (закрывает единственный реальный gap)
**Проблема:** новый файл `migrations/020+` может сделать `ALTER TABLE reconciliation_*` и обойти:
- Hash Lock (он не трогает 016–019),
- Boundary Grep (он про Python),
- Ruff (не про SQL).

**Решение:** в CI запрещаем любые упоминания `reconciliation_audit_log|reconciliation_alerts|reconciliation_suppressions` в миграциях `020+`.  
Это “грубый”, но корректный запрет: легитимного случая упоминать эти таблицы в новых миграциях нет.

---

### M5 — PROJECT_DNA + AI Reviewer Prompt (контекст + контроль качества)
**PROJECT_DNA.md**: короткий файл “как устроен проект, что нельзя делать, словарь терминов”.  
**AI Reviewer Prompt**: один промпт для роли “ревьюер”, который:
- ищет нарушение границ (Tier‑1/Tier‑2/Tier‑3),
- ищет idempotency/trace_id/commit‑в‑не‑том‑месте,
- ищет утечки секретов/PII в логах.

**Покрывает:** снижение галлюцинаций и ошибок “с первого раза”, дешёвое качество.

---

### M6 — DEFINITION_OF_DONE (чек‑лист на PR/таску)
8–12 пунктов, которые закрывают типовые “болячки”:
- trace_id есть?
- idempotency key есть?
- нет прямых DML на core tables в Tier‑3?
- нет коммита внутри доменной операции?
- тесты/CI зелёные?
- логирование без секретов?

**Покрывает:** дисциплина конвейера, снижает дебаг и возвраты.

## 3) Проверка на логические противоречия (формально)

**Противоречий нет**, если соблюдены 4 правила:

1) **Hash Lock** применяется только к Tier‑1 списку, который стабилен и опубликован (в `docs/PHASE2_BOUNDARY.md`).  
2) **Boundary Grep** не должен пытаться “понимать логику” — только границы и запреты.  
3) **Ruff** вводится мягко (иначе CI станет “красной стеной”).  
4) **Migration DDL Guard** смотрит только `migrations/020+`, иначе он начнёт блокировать сам Core.

## 7) Приложение: 1‑страничный итог (для вставки в Master Plan)

**Post‑Core Acceleration (Minimal, No Overengineering)**  
Мы ускоряем разработку Tier‑3 автоматизаций без деградации качества за счёт 6 механизмов:

1) Tier‑1 Hash Lock (CRLF‑safe)  
2) Ruff + pre‑commit (warn‑only baseline)  
3) Boundary Grep: запрет DML на reconciliation_* и core business tables из Tier‑3  
4) Migration DDL Guard: запрет упоминаний reconciliation_* в migrations/020+  
5) PROJECT_DNA + AI Reviewer Prompt  
6) DEFINITION_OF_DONE (PR checklist)

Эти механизмы не меняют Core, не требуют постоянной поддержки и дают “конвейер”: новые автоматизации проходят автоматические проверки до того, как их увидит founder.
