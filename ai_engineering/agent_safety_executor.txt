# Auto-Agent Safety Executor (Always-On Mode)

Эти правила применяются ко всем планам перед выполнением в режиме AGENT.

## 1. Проверка структуры плана
- План должен состоять только из пронумерованных шагов.
- Каждый шаг — атомарен (один файл, одно действие).
- Не допускаются общие формулировки ("исправить", "оптимизировать", "обновить модуль").

## 2. Проверка безопасных действий
Запрещено выполнять шаги, содержащие:
- массовые изменения,
- работу с директориями,
- удаление файлов или папок,
- создание больших структур,
- модификацию неизвестных файлов,
- изменение кода или конфигурации вне явно указанных файлов.

## 3. Проверка Controlled Context Mode
- Шаги не должны ссылаться на файлы или пути, которые пользователь не перечислил.
- Запрещены действия, требующие анализа проекта или чтения директорий.
- Недопустимо выполнять шаги, выходящие за рамки плана.

## 4. Проверка архитектурных принципов
Каждый шаг должен соответствовать:
- ARCHITECTURE_GUIDE.md,
- CONTEXT_RULES.md,
- agent_behavior_rules.txt,
- AI_WORKSTYLE_GUIDE.md,
- **MECHANISM_HIERARCHY.md (ОБЯЗАТЕЛЬНО):**
  - Если шаг создаёт новый механизм (validation, safety, guard, protocol, enforcement и т.д.) — Executor ОБЯЗАН проверить MECHANISM_HIERARCHY.md.
  - Если PRIMARY механизм для области логики уже существует — шаг БЛОКИРУЕТСЯ с пометкой "Architectural Error: PRIMARY mechanism exists in MECHANISM_HIERARCHY.md".
  - Нарушение этого правила считается архитектурной ошибкой и блокирует выполнение плана.

## 5. Проверка на риск
Шаг считается рискованным, если:
- он изменяет много строк,
- он затрагивает важный модуль без объяснения,
- он вызывает внешние процессы,
- он предлагает неявные изменения.

Рискованные шаги должны быть:
- либо заблокированы,
- либо помечены как "Requires explicit confirmation".

## 6. Fail-Safe Mode
Если план нарушает хотя бы одно правило:
- Executor блокирует выполнение,
- указывает, какие шаги нарушают правила,
- предлагает безопасный переформулированный вариант.

## 7. Принцип прозрачности
Перед выполнением изменений Executor обязан:
- вывести итоговый очищенный план,
- дождаться подтверждения пользователя.
