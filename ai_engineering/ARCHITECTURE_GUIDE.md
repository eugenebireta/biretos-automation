# 🏛 Architecture Guide & Engineering Principles

Этот документ фиксирует инженерные стандарты и архитектурные ограничения проекта **Biretos Automation**.

## 18 инженерных принципов (критичные правила программирования)

**ПРИМЕЧАНИЕ:** Эти 18 принципов также известны как "18 критичных правил программирования" проекта. Это один и тот же набор принципов, зафиксированный в этом документе.

### Критические (Foundation)
1. **Идемпотентность** — повторный запуск операции не меняет результат и не дублирует данные.
2. **Атомарность** — действие выполняется целиком или откатывается, промежуточные состояния недопустимы.
3. **Graceful Degradation** — при отказе подсистемы проект продолжает работу в урезанном режиме.
   - **СТАТУС:** DECLARED (декларативный принцип, не требует автоматического enforcement)
   - **ПРИМЕНЕНИЕ:** Реализуется на уровне архитектуры системы, не через механизмы проверки

### Архитектурные (Core)
4. **Single Responsibility** — один скрипт выполняет одну задачу.
5. **Explicit > Implicit** — явные аргументы и абсолютные пути вместо скрытой магии.
6. **Fail Fast & Loud** — ошибки проявляются сразу и логируются громко.
7. **Structured Logging** — логи читаемы человеком и пригодны для машинной обработки.
8. **Security First** — секреты только в `.env`, минимум доверия внешним системам.
9. **KISS** — избегать лишней сложности.
10. **Modular Isolation** — изменения в модуле не ломают соседние.

### Процессные (Process)
11. **Human-Readable Code** — код и комментарии ориентированы на инженера и AI.
12. **Defense in Depth** — валидация входных данных на каждом слое.
13. **Immutability Priority** — минимизировать глобальное состояние.
14. **Documented Decisions** — любые архитектурные компромиссы фиксируются.
15. **Testability** — код проектируется с учетом возможности тестирования.
16. **Performance Awareness** — избегать блокирующих паттернов (например, PowerShell pipes).
17. **Standardized Environment** — единые пути, версии, окружения.
18. **Continuous Refactoring** — поддержка чистоты кода по правилу Boy Scout.
   - **СТАТУС:** DECLARED (декларативный принцип, не требует автоматического enforcement)
   - **ПРИМЕНЕНИЕ:** Реализуется на уровне практики разработки, не через механизмы проверки

## Core Project Rules

### Canonical Logic First — Запрет дублирования

**КРИТИЧЕСКОЕ ПРАВИЛО:** Приоритет канонической логики над созданием нового. Любое дублирование логики, скриптов, модулей или workflow **ЗАПРЕЩЕНО**.

#### Принцип

**Перед созданием нового решения ОБЯЗАТЕЛЬНО:**
1. Проверить существующие скрипты, модули, workflow в проекте
2. Использовать существующее решение или расширить его
3. Создавать новое ТОЛЬКО если существующее решение принципиально неприменимо

#### Запрещённые действия

- **Дублирование скриптов:** создание нового скрипта с логикой, аналогичной существующему
- **Дублирование модулей:** создание нового модуля/класса с функциональностью, уже реализованной в проекте
- **Дублирование workflow:** создание нового n8n workflow, дублирующего логику существующего
- **Альтернативные реализации:** создание "улучшенной версии" без удаления/замены старой
- **Параллельные решения:** существование нескольких решений одной задачи

#### Обязательные действия

- **Reuse существующего:** использовать существующий скрипт/модуль/workflow
- **Расширение существующего:** добавлять функциональность в существующее решение
- **Рефакторинг существующего:** улучшать существующее решение вместо создания нового
- **Документирование reuse:** явно указывать, какое существующее решение используется

#### Допустимые исключения

**Создание нового решения разрешено ТОЛЬКО в случаях:**

1. **Принципиальная несовместимость:**
   - Существующее решение технически не может быть расширено для новой задачи
   - Требуется принципиально иная архитектура (например, синхронный vs асинхронный подход)
   - Доказательство несовместимости должно быть задокументировано

2. **Изоляция по требованию:**
   - Новое решение требуется для изолированного проекта/подсистемы
   - Изоляция является архитектурным требованием (например, `brand-catalog-automation/`)
   - Изоляция явно задокументирована в архитектуре проекта

3. **Временный эксперимент:**
   - Создание экспериментального решения в `_scratchpad/` для проверки гипотезы
   - Эксперимент имеет явный срок жизни и план миграции в каноническое решение
   - После эксперимента решение либо удаляется, либо интегрируется в каноническое

#### Связь с другими правилами

- **AI-механизмы:** подчиняются `MECHANISM_HIERARCHY.md` (Reuse-First Rule для AI-механизмов)
- **Shopware API:** используют канонические pipeline (см. `UPDATE_ARCHITECTURE_CHANGES.md`)
- **SSH/SCP:** используют канонические обёртки (`Invoke-SafeSsh.ps1`, `Invoke-SafeScp.ps1`)
- **Background processes:** используют канонические скрипты (`infrastructure/safe-run/*.ps1`)

#### Статус нарушения

**Нарушение этого правила считается архитектурной ошибкой** и требует:
- Отказа от создания дублирующего решения
- Использования или расширения существующего канонического решения
- Документирования выбора существующего решения

## Архитектурные ограничения

### PowerShell SSH/SCP Deadlock
- **Проблема:** стандартные pipes в PowerShell блокируются при больших объёмах вывода `ssh`/`scp`.
- **Решение:** использовать файловые буферы (реализация в `VPSRunner.psm1`), не применять `WaitForExit` без таймаутов.

### SSH/VPS Interaction Protocol
- **Нельзя:** передавать сложные Bash-команды (с `&&`, пайпами, циклами, `${VAR}`) напрямую через PowerShell/SSH.
- **Можно:** простые one-liner (например, `ls`, `cat`, `uptime`).
- **Обязательный метод:** для всего сложнее — писать скрипт `.sh` локально (`_scratchpad/`), загружать через `scp` и запускать через `ssh`.
- **Причина:** предотвращение Escaping Hell и зависаний stdin/stdout.

### Multi-Server Management
- Конфигурация всех серверов хранится в `infrastructure/config/servers.json`.
- SSH/SCP операции выполняются через обёртки `Invoke-SafeSsh.ps1` и `Invoke-SafeScp.ps1` (таймауты, логирование).
- Прямые вызовы `ssh.exe` допустимы только для чтения (`ls`, `cat`) и с флагами безопасности.
- Все попытки фиксируются в `_scratchpad/ssh_activity.log` (START/SUCCESS/TIMEOUT).

### Принцип Чистого Корня
- **Проблема:** временные файлы в корне мешают навигации.
- **Решение:** хранить `tmp_*` и аналогичные артефакты в `_scratchpad/`, поддерживать чистоту корневой директории.
- **СТАТУС:** Операционный принцип (не enforced-механизм)
- **ПРИМЕНЕНИЕ:** 
  - Временные файлы допустимы в процессе работы
  - Подлежат периодической очистке (через `infrastructure/scripts/Clean-Workspace.ps1` или вручную)
  - Принцип направлен на поддержание чистоты, а не на автоматическую блокировку создания временных файлов
